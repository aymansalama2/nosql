Partie 1: Comparaison SQL vs NoSQL

Introduction

Les bases de données constituent le cœur de tout système d'information moderne. Avec l'évolution des besoins technologiques, deux grandes familles de systèmes de gestion de bases de données (SGBD) se sont imposées : les bases relationnelles (SQL) et les bases non-relationnelles (NoSQL).

1.1 Bases de Données Relationnelles (SQL)

Définition

Les bases de données relationnelles organisent les données sous forme de tables liées entre elles par des relations. Elles utilisent le langage SQL (Structured Query Language) pour manipuler les données.

MySQL - Exemple Populaire

MySQL est l'un des SGBD relationnels les plus utilisés au monde. Open source, rapide et fiable, il est parfait pour les applications web et les systèmes d'entreprise.

Caractéristiques Principales

  • Structure rigide avec schéma fixe
  • Propriétés ACID (Atomicité, Cohérence, Isolation, Durabilité)
  • Langage SQL standardisé
  • Relations entre tables (clés étrangères)
  • Intégrité référentielle
  • Transactions complexes

1.2 Bases de Données Non-Relationnelles (NoSQL)

Définition et Évolution

NoSQL signifie "Not Only SQL". Ces bases de données sont apparues pour répondre aux limitations des bases relationnelles face aux besoins du Big Data, de la scalabilité horizontale et de la flexibilité des schémas.

Document

MongoDB, CouchDB

Clé-Valeur

Redis, DynamoDB

Colonnes

Cassandra, HBase

Graphe

Neo4j, ArangoDB

Partie 2: Solutions de TP

2.1 TP1 - Conception d'une Base de Données Relationnelle

Énoncé

Concevoir une base de données pour un système de gestion de bibliothèque universitaire. Le système doit gérer les livres, les auteurs, les étudiants, les emprunts et les réservations.

Solution - Modèle Conceptuel

Diagramme Entité-Relation de la bibliothèque
ENTITÉS:
- LIVRE (id_livre, titre, isbn, année_publication, nb_exemplaires)
- AUTEUR (id_auteur, nom, prénom, nationalité, date_naissance)
- ÉTUDIANT (id_étudiant, nom, prénom, email, filière, année_étude)
- EMPRUNT (id_emprunt, date_emprunt, date_retour_prévue, date_retour_réelle)
- RÉSERVATION (id_réservation, date_réservation, statut)

RELATIONS:
- ÉCRIRE (AUTEUR, LIVRE) - N:M
- EMPRUNTER (ÉTUDIANT, LIVRE, EMPRUNT) - 1:N:M
- RÉSERVER (ÉTUDIANT, LIVRE, RÉSERVATION) - 1:N:M

Solution - Script SQL

-- Création des tables
CREATE TABLE auteurs (
    id_auteur INT PRIMARY KEY AUTO_INCREMENT,
    nom VARCHAR(100) NOT NULL,
    prénom VARCHAR(100),
    nationalité VARCHAR(50),
    date_naissance DATE
);

CREATE TABLE livres (
    id_livre INT PRIMARY KEY AUTO_INCREMENT,
    titre VARCHAR(200) NOT NULL,
    isbn VARCHAR(13) UNIQUE,
    année_publication YEAR,
    nb_exemplaires INT DEFAULT 1
);

CREATE TABLE étudiants (
    id_étudiant INT PRIMARY KEY AUTO_INCREMENT,
    nom VARCHAR(100) NOT NULL,
    prénom VARCHAR(100),
    email VARCHAR(150) UNIQUE,
    filière VARCHAR(100),
    année_étude INT
);

CREATE TABLE emprunts (
    id_emprunt INT PRIMARY KEY AUTO_INCREMENT,
    id_étudiant INT,
    id_livre INT,
    date_emprunt DATE NOT NULL,
    date_retour_prévue DATE NOT NULL,
    date_retour_réelle DATE,
    FOREIGN KEY (id_étudiant) REFERENCES étudiants(id_étudiant),
    FOREIGN KEY (id_livre) REFERENCES livres(id_livre)
);

2.2 TP2 - Base de Données NoSQL (MongoDB)

Énoncé

Implémenter un système de blog avec MongoDB. Gérer les articles, commentaires, catégories et statistiques de lecture avec une structure de documents flexible.

Solution - Structure des Documents

// Collection: articles
{
  "_id": ObjectId("..."),
  "titre": "Introduction aux bases NoSQL",
  "contenu": "Le contenu de l'article...",
  "auteur": {
    "nom": "Dupont",
    "email": "dupont@email.com"
  },
  "catégories": ["NoSQL", "Bases de données", "MongoDB"],
  "date_publication": ISODate("2025-11-16"),
  "statistiques": {
    "vues": 150,
    "likes": 23,
    "partages": 5
  },
  "commentaires": [
    {
      "auteur": "Martin",
      "contenu": "Très bon article!",
      "date": ISODate("2025-11-16")
    }
  ],
  "tags": ["nosql", "mongodb", "database"]
}

Solution - Requêtes MongoDB

// Insérer un article
db.articles.insertOne({
  titre: "Guide MongoDB",
  contenu: "Contenu détaillé...",
  auteur: { nom: "Alice", email: "alice@test.com" },
  catégories: ["NoSQL", "Tutorial"],
  date_publication: new Date(),
  statistiques: { vues: 0, likes: 0, partages: 0 },
  commentaires: [],
  tags: ["mongodb", "guide"]
});

// Rechercher articles par catégorie
db.articles.find({ "catégories": "NoSQL" });

// Mettre à jour les statistiques
db.articles.updateOne(
  { "_id": ObjectId("...") },
  { $inc: { "statistiques.vues": 1 } }
);

// Ajouter un commentaire
db.articles.updateOne(
  { "_id": ObjectId("...") },
  { $push: { 
      "commentaires": {
        auteur: "Bob",
        contenu: "Excellent!",
        date: new Date()
      }
    }
  }
);

Conclusion

Ce rapport a exploré les différences fondamentales entre les bases de données relationnelles (SQL) et non-relationnelles (NoSQL), en mettant l'accent sur MySQL comme exemple de SGBD relationnel.

Synthèse des apprentissages :

Forces du SQL

  • • Intégrité des données garantie
  • • Transactions ACID complètes
  • • Langage standardisé et mature
  • • Excellent pour les données structurées

Avantages du NoSQL

  • • Scalabilité horizontale native
  • • Flexibilité des schémas
  • • Performance sur gros volumes
  • • Adapté aux données semi-structurées

Les travaux pratiques ont démontré l'importance de choisir la technologie appropriée selon le contexte. Une bibliothèque universitaire bénéficie de la structure rigide du SQL, tandis qu'un blog moderne tire parti de la flexibilité de MongoDB.

"Le choix entre SQL et NoSQL n'est pas une question de supériorité, mais d'adéquation aux besoins spécifiques du projet."
